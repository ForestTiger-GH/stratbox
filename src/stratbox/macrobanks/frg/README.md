# FRG (`stratbox.macrobanks.frg`)

Модуль `stratbox.macrobanks.frg` предназначен для первого этапа работы с внешними файлами FRG.
На текущем этапе модуль **не читает содержимое Excel-файлов** и **не выполняет парсинг таблиц**. Его задача — стабильно привести файловый контур в порядок перед дальнейшей предметной обработкой.

Модуль решает четыре практические задачи:

1. считывает список файлов из каталога верхнего уровня;
2. распознаёт семейство файла по имени и нормализует период;
3. выбирает наиболее актуальный файл по каждому семейству;
4. при необходимости переименовывает, копирует, удаляет лишние файлы и собирает итоговый ZIP-архив.

---

## Ключевые принципы

### 1. Модуль работает только по именам файлов

Внутрь Excel-файлов модуль не заходит:

- не читает листы;
- не меняет формат;
- не меняет содержимое;
- не делает пересохранение книги как Excel-объекта.

Все решения принимаются только на основе:

- имени файла;
- расширения;
- пути;
- базовых файловых метаданных.

### 2. Обход идёт только по верхнему уровню каталога

Это сделано специально, чтобы не затрагивать вложенные каталоги, например папку `Архив`.

По умолчанию:

- вложенные папки не сканируются;
- вложенные папки не удаляются;
- содержимое вложенных папок не архивируется.

### 3. Нераспознанные файлы не трогаются

Если файл:

- имеет случайное имя;
- назван с ошибкой;
- относится к другому процессу;
- не подходит ни под одну поддерживаемую схему имени;

то модуль оставляет его как есть.

Такие файлы:

- не переименовываются;
- не удаляются;
- не считаются ошибкой выполнения.

Это правило принципиально важно для безопасной работы в общей папке.

### 4. Поддерживаются два формата имён

Модуль умеет распознавать:

1. **исходные имена FRG**;
2. **внутренние имена**, которые уже сформировала сама библиотека после зачистки.

Это позволяет запускать обработку повторно в одном и том же каталоге, где рядом могут лежать:

- новые поставленные файлы FRG;
- уже переименованные итоговые файлы предыдущего запуска.

При совпадении семейства и периода приоритет отдаётся **внутреннему стандартному имени**.

## Поддерживаемые семейства файлов

На текущем этапе предусмотрены следующие семейства:

- `express_issuance` — Экспресс выдачи
- `express_portfolios` — Экспресс портфели
- `express_passives` — Экспресс — Пассивы
- `express_issuance_weekly` — Экспресс выдачи — Weekly
- `express_issuance_cards` — Экспресс выдачи — Карты
- `regional_issuance` — Региональные выдачи
- `regional_portfolios` — Региональные портфели
- `regional_issuance_mortgage` — Региональные выдачи — Ипотека
- `regional_portfolios_mortgage` — Региональные портфели — Ипотека
- `regional_issuance_cards` — Региональные выдачи — Карты
- `rbm_volumes_q` — Выдачи — Q
- `rbm_portfolios_q` — Портфели — Q
- `volumes_cards_q` — Выдачи — Карты — Q
- `mortgage_refinancing` — Рефинансирование Ипотеки

Порядок распознавания задаётся в `registry.py`: сначала идут более специфичные семейства, затем более общие.

## Нормализация периода

Модуль извлекает период из имени файла и приводит его к стандартному виду.

### Календарные даты

Приводятся к формату:

- `YYYY-MM-DD`

Распознаются даты с разделителями:

- `.`
- `-`
- `_`
- `/`

и их смешанные комбинации.

### Недельные периоды

Недельные файлы приводятся к формату:

- `YYYY-MM week N`

Примеры эквивалентных входных вариантов:

- `Май 2026 1 неделя`
- `2026 Февраль 3 неделя`
- `Выдачи - Май 2026 2 недели`

Итоговая нормализация будет, например:

- `2026-05 week 1`
- `2026-02 week 3`

---

## Правило по weekly-файлам

Для weekly-файлов действует отдельная логика актуальности.

Если внутри одного и того же месяца присутствуют:

- `week 1`
- `week 2`
- `week 3`

то более ранние недели считаются устаревшими.

Например:

- `2026-05 week 1` устаревает, если есть `2026-05 week 2`;
- `2026-05 week 2` устаревает, если есть `2026-05 week 3`.

Такие файлы получают причину:

- `superseded_by_newer_week_in_same_month`

и в режиме зачистки могут удаляться как неактуальные.

---

## Правило по региональным данным

Для региональных файлов в коде зафиксирована отсечка новой схемы начиная с:

- `2026-04-01`

Это означает:

- более старые региональные файлы видны в полном каталоге;
- но они не попадают в `latest`;
- в режиме зачистки их можно удалить как слишком старые по правилу семейства.

Причина такой невалидности:

- `period_older_than_family_min_date`

---

## Внутренняя схема имён файлов

После зачистки библиотека может создать рядом с исходными файлами свои нормализованные имена.

Сейчас используется активная схема:

- `период_подпись.ext`

Примеры:

- `2026-05-01_Экспресс выдачи.xlsx`
- `2026-05 week 2_Экспресс выдачи — Weekly.xlsx`
- `2026-05-15_Экспресс — Пассивы.xlsb`

Схема задаётся централизованно в модуле:

- `filename_scheme.py`

Если в будущем потребуется добавить префикс, например:

- `FRG_2026-05-01_Экспресс выдачи.xlsx`

это нужно менять именно там.

Важно:

- поддерживается **одна активная внутренняя схема**;
- старая внутренняя схема специально не хранится в виде отдельного набора обратной совместимости;
- если стандарт имени меняется, это должно делаться как отдельная управляемая миграция.

## Основной публичный API

### 1. Сканирование каталога и выбор актуальных файлов

```python
from stratbox.macrobanks.frg import run_frg_stage1

result = run_frg_stage1("/path/to/frg_files")
catalog_df = result["catalog"]
latest_df = result["latest"]
dispatch_df = result["dispatch"]
```

Что возвращается:

- `catalog` — полный каталог найденных файлов;
- `latest` — по одному наиболее актуальному файлу на семейство;
- `dispatch` — журнал отправки актуальных файлов в заглушки будущих парсеров.

### 2. Построение каталога отдельно

```python
from stratbox.macrobanks.frg import build_frg_catalog

catalog_df = build_frg_catalog("/path/to/frg_files")
```

### 3. Отбор актуальных файлов отдельно

```python
from stratbox.macrobanks.frg import select_latest_frg_files

latest_df = select_latest_frg_files(catalog_df)
```

### 4. Зачистка каталога

```python
from stratbox.macrobanks.frg import run_frg_cleanup

result = run_frg_cleanup(
    "/path/to/frg_files",
    delete_others=False,
    archive_latest=False,
    execute=False,
)

catalog_df = result["catalog"]
latest_df = result["latest"]
plan_df = result["plan"]
execution_df = result["execution"]
```

---

## Что делает зачистка каталога

### Режим без удаления (`delete_others=False`)

Модуль:

- определяет `latest` по каждому семейству;
- строит новое стандартное имя;
- при `execute=True` создаёт копию актуального файла рядом в том же каталоге.

Исходные файлы при этом остаются на месте.

### Режим с удалением (`delete_others=True`)

Модуль:

- переименовывает актуальные файлы в стандартный внутренний вид;
- удаляет другие распознанные, но неактуальные файлы;
- удаляет weekly-файлы, вытесненные более поздней неделей того же месяца;
- удаляет региональные файлы, признанные слишком старыми по правилу семейства.

При этом модуль **не удаляет**:

- нераспознанные файлы;
- случайные файлы в каталоге;
- вложенные каталоги;
- содержимое вложенных каталогов.

## Архивирование итоговых файлов

В зачистку встроена отдельная опция архивирования.

Пример:

```python
from stratbox.macrobanks.frg import run_frg_cleanup

result = run_frg_cleanup(
    "/path/to/frg_files",
    delete_others=True,
    archive_latest=True,
    execute=True,
)
```

Что делает эта опция:

- создаёт ZIP-архив только с итоговыми `latest`-файлами;
- сохраняет архив в тот же каталог верхнего уровня;
- не заходит во вложенные папки;
- если архив уже существует, полностью его перезаписывает.

Имя архива формируется так:

- `FRG_Actuals_YYYY-MM-DD.zip`

Пример:

- `FRG_Actuals_2026-05-26.zip`

Архив не используется для распознавания и не участвует в отборе `latest`.

## Что лежит в модуле

- `api.py` — публичный API
- `models.py` — модели правил и записей каталога
- `registry.py` — реестр семейств FRG
- `naming.py` — распознавание исходных имён поставщика и внутренних имён библиотеки
- `filename_scheme.py` — активная схема внутренних имён
- `catalog.py` — построение полного каталога
- `selection.py` — выбор наиболее актуальных файлов
- `dispatch.py` — маршрутизация в заглушки парсеров
- `cleanup.py` — план и выполнение зачистки каталога, включая архивирование
- `parsers/` — заглушки будущих парсеров по группам семейства
- `run_scan_debug.py` — служебный запуск для отладки

---

## Практическая роль текущего этапа

Текущий этап — это **подготовка стабильного файлового слоя** перед реальным предметным парсингом.

Иными словами, модуль уже умеет:

- привести каталог в контролируемое состояние;
- оставить только актуальный набор файлов;
- дать стабильные имена для дальнейших ссылок и downstream-обработки;
- безопасно работать повторно в одном и том же каталоге.

А следующий этап развития — это уже полноценные парсеры содержимого Excel-файлов по каждому семейству.
