# `stratbox.macrobanks`

`stratbox.macrobanks` — доменный слой библиотеки `stratbox` для задач на стыке макроэкономики, банковской статистики, отчетных форм, файловых поставок и подготовки рабочих витрин.

Каталог объединяет несколько самостоятельных доменов. Каждый домен решает отдельную прикладную задачу, но опирается на общую инфраструктуру `stratbox.base`:

- `FileStore` для чтения и записи файлов;
- `ioapi` для форматных операций;
- `base.net` для скачивания внешних источников;
- `common.time` для генерации дат;
- встроенные справочники из `registries`, если они нужны домену.

## Домены

```text
macrobanks/
  cbr_forms/     # отчетные формы Банка России: 101, 102, 123, 135, 805
  escrow/        # счета эскроу: скачивание, парсинг, long/pivot, Excel/ZIP
  frg/           # файловый контур FRG: каталог, latest, зачистка, архивирование
  cbr_file_collector/  # скачивание статистических файлов Банка России без обработки содержимого
```

## Публичные входы

### CBR forms

```python
from stratbox.macrobanks.cbr_forms import run_all_forms_to_xlsx

out = run_all_forms_to_xlsx(
    date_from="2024-01-01",
    date_to="2024-03-01",
    forms="101,102,135,805",
    out_dir="outputs/cbr_forms",
)
```

### Escrow

```python
from stratbox.macrobanks.escrow import run_escrow_to_xlsx

result = run_escrow_to_xlsx(
    out_path="outputs/Escrow Accounts.xlsx",
    source_cache_dir="cache/escrow",
)
```

### FRG

```python
from stratbox.macrobanks.frg import run_frg_stage1, run_frg_cleanup

scan = run_frg_stage1("data/frg")
cleanup = run_frg_cleanup("data/frg", archive_latest=True, execute=False)
```

### CBR archiver

```python
from stratbox.macrobanks.cbr_file_collector import CbrFileCollectRequest, collect_cbr_files

result = collect_cbr_files(
    CbrFileCollectRequest(
        target_path="/content/CBR Collected Files.zip",
        save_mode="zip",
    )
)
```

## Общие принципы доменного слоя

### 1. Отделение предметной логики от среды исполнения

Доменам не нужно знать, работает код в Colab, локально или внутри корпоративного контура. Для файловых операций используется `FileStore`, для скачивания — `base.net`.

### 2. Сначала поток данных, потом витрина

Для доменов, которые действительно обрабатывают таблицы, предпочтительный подход такой:

1. получить исходник;
2. разобрать его в нормализованный поток данных;
3. объединить, отфильтровать, отсортировать и проверить поток;
4. только после этого формировать wide/pivot/Excel-витрину.

Именно этот принцип применяется в `cbr_forms` и `escrow`.

### 3. Исходные файлы не меняются без явной необходимости

Если домен занимается файловым контуром или архивированием, он не должен пересохранять Excel и менять содержимое файлов. Например:

- `frg` работает по именам файлов и не открывает Excel-книги;
- `cbr_file_collector` скачивает исходные файлы ЦБ и сохраняет их как есть.

### 4. Реестры должны быть явными

Правила распознавания, списки источников и перечни показателей должны храниться в явных реестрах:

- формы и показатели — в `cbr_forms`;
- FRG-семейства — в `frg/registry.py`;
- ссылки на статистические файлы ЦБ — в `cbr_file_collector/registry.py`;
- показатели escrow — в соответствующем реестре домена.

Это позволяет сопровождать библиотеку вручную и безопасно, без скрытой магии в середине конвейера.

## Что возвращают доменные функции

Доменный API должен возвращать не только факт сохранения файла, но и проверяемый результат выполнения:

- путь к итоговому файлу или папке;
- список обработанных источников;
- список ошибок;
- таблицы каталога, latest или плана;
- служебные счетчики.

Это особенно важно для Jupyter Notebook: пользователь должен быстро понять, что было сделано и что требует внимания.
