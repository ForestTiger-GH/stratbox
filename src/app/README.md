# app

`app` — это пользовательская desktop-оболочка Strategy Box.

Этот пакет расположен **рядом** со `stratbox`, а не внутри него, потому что `stratbox` является библиотекой и прикладным движком, а `app` — отдельным пользовательским слоем поверх него.

Идея простая:

- `stratbox` содержит доменную и инфраструктурную логику;
- `app` показывает задачи, профили среды, параметры запуска, статус, логи и результаты;
- будущий `Launcher.exe` будет подготавливать среду и запускать `python -m app`.

На текущем этапе `app` — это **первая архитектурная версия приложения**: рабочий каркас, в который дальше подключаются реальные сценарии и сервисные возможности.

---

## Зачем нужен `app`

До появления `app` репозиторий использовался в основном как библиотека и набор сценариев, запускаемых из notebook или скриптов.

`app` добавляет к этому пользовательский слой:

- выбор профиля данных;
- диагностику среды;
- реестр задач;
- единый способ запуска сценариев;
- журнал выполнения;
- отображение версии и состояния локального репозитория.

`app` **не должен** содержать предметную бизнес-логику обработки данных. Его задача — собирать пользовательские параметры, запускать готовые модули `stratbox` и показывать результат.

---

## Граница ответственности

### За что отвечает `app`

`app` отвечает за:

- запуск пользовательского интерфейса;
- чтение пользовательского конфига;
- работу с профилями среды;
- создание активного `FileStore` для текущей сессии;
- загрузку реестра задач;
- запуск задач через adapter-слой;
- вывод статуса, логов и результатов.

### За что отвечает `stratbox`

`stratbox` отвечает за:

- файловую инфраструктуру (`FileStore`, `ioapi`, runtime);
- загрузку и сохранение данных;
- доменные сценарии (`cbr_archiver`, `frg`, `escrow`, `cbr_forms` и т.д.);
- справочники, регистры и доменные преобразования;
- прикладную логику, не зависящую от GUI.

### Главное правило зависимости

Зависимость разрешена только в одну сторону:

```text
app -> stratbox
```

Обратной зависимости быть не должно.

`app` может импортировать `stratbox`, но `stratbox` не должен импортировать `app`.

Это фундаментальное правило архитектуры.

## Структура каталога

```text
app/
    __init__.py
    __main__.py

    core/
    gui/
    profiles/
    tasks/
    resources/
```

### `__main__.py`

Точка входа приложения.

Позволяет запускать:

```bash
python -m app
```

Также поддерживает сервисный режим:

```bash
python -m app --diagnose
```

### `core/`

Базовый служебный слой приложения:

- пути;
- пользовательский конфиг;
- логирование;
- сведения о версии;
- сборка `AppContext`.

Здесь приложение понимает:

- где находится репозиторий;
- где хранить пользовательские настройки;
- где лежат логи;
- какой профиль выбран;
- какой `FileStore` активен;
- какую версию локального репозитория оно сейчас использует.

### `profiles/`

Слой файловой среды.

Профиль отвечает на вопрос:

> где находятся данные в текущей сессии?

Именно здесь определяется корневой путь данных и создаётся активный `FileStore`.

На первом этапе профили считаются локальными и работают через `LocalFileStore`.

### `tasks/`

Слой задач приложения.

Здесь находятся:

- модели задач;
- реестр задач;
- runner;
- adapters.

GUI не запускает доменные модули напрямую. Он работает через `TaskSpec` и adapter-слой.

### `gui/`

Интерфейс приложения.

Здесь находятся:

- запуск Qt-интерфейса;
- главное окно;
- фоновые worker-задачи.

GUI отвечает только за отображение, сбор параметров и запуск задач.

### `resources/`

Встроенные ресурсы приложения:

- профили;
- JSON-конфиги задач;
- стили;
- будущие иконки и изображения.

---

## Точка входа

Основной запуск:

```bash
python -m app
```

Сервисный запуск без GUI:

```bash
python -m app --diagnose
```

Консольный script после установки:

```bash
stratbox-app
```

Идея такая:

- в разработке приложение запускается прямо из репозитория;
- позже `Launcher.exe` будет запускать это же приложение автоматически.

---

## Требования к окружению

Для GUI требуется extra-зависимость `app`.

Рекомендуемая установка в editable-режиме:

```bash
pip install -e ".[app]"
```

Почему это важно:

- `stratbox` как библиотека может использоваться без GUI;
- `PySide6` и связанные GUI-зависимости не должны быть обязательными для всех сценариев работы с репозиторием.

Если GUI-зависимости не установлены, приложение должно оставаться запускаемым в сервисном режиме и выдавать понятное сообщение.

---

## Пользовательский конфиг

`app` хранит пользовательские настройки **вне репозитория**.

Репозиторий обновляется через Git, поэтому пользовательские настройки не должны храниться внутри `src/` или рядом с кодом.

Текущий пользовательский конфиг:

```text
%LOCALAPPDATA%\Stratbox\config\app.json
```

fallback для не-Windows среды:

```text
~/.local/share/Stratbox/config/app.json
```

В этом конфиге хранятся настройки самого GUI:

- активный профиль;
- последняя выбранная задача;
- размер окна.

Важно: это **не launcher-конфиг**.

Launcher в будущем будет хранить свои bootstrap-настройки отдельно.

---

## Профили среды

Профиль описывает файловую среду приложения.

На первом этапе поддерживаются профили вида:

- `local_c`;
- `netdrive_w`;
- `home_server`.

Они задаются в:

```text
app/resources/profiles/default_profiles.json
```

Профиль содержит:

- `id` — техническое имя;
- `title` — отображаемое имя;
- `kind` — тип файловой среды;
- `root` — корневой путь данных;
- `required_dirs` — обязательные подкаталоги;
- `description` — описание;
- `readonly` — признак режима только чтения.

### Текущий принцип работы

Пока все профили считаются локальными и создают:

```text
LocalFileStore(root=...)
```

Это означает, что:

- локальная папка на `C:` работает как обычное локальное хранилище;
- NetDrive `W:` на текущем этапе тоже рассматривается как смонтированный локальный диск, а не как отдельный транспорт.

Это сознательное решение первой версии приложения.

---

## Диагностика профиля

Для активного профиля доступна проверка:

- существует ли корень;
- является ли корень директорией;
- можно ли читать содержимое;
- можно ли писать тестовый файл;
- существуют ли обязательные каталоги.

Диагностика нужна для того, чтобы пользователь видел проблему среды до запуска прикладной задачи, а не после падения сценария.

На первом этапе это один из важнейших сервисных механизмов.

---

## Реестр задач

Задачи приложения задаются не в коде окна, а через JSON-описания.

Конфиги лежат в:

```text
app/resources/tasks/
```

Это позволяет:

- добавлять задачи без переписывания GUI;
- обновлять список задач обычным обновлением репозитория;
- отделить пользовательское описание задачи от внутреннего Python-кода.

Каждый конфиг задачи описывает:

- `id`;
- `title`;
- `description`;
- `category`;
- `adapter`;
- `params`;
- `input_dir`;
- `output_dir`;
- `links`;
- `enabled`.

---

## Почему нужен adapter-слой

GUI не должен напрямую вызывать сложные функции `stratbox`.

Вместо этого используется промежуточная прослойка:

```text
GUI -> TaskSpec -> adapter -> stratbox
```

Adapter нужен для того, чтобы:

- перевести пользовательские параметры в формат доменной функции;
- изолировать GUI от внутренней сигнатуры `stratbox`;
- не смешивать интерфейсные решения и прикладной код.

Это один из ключевых принципов `app`.

## Текущие задачи

### `environment_check`

Сервисная задача.

Проверяет:

- активный профиль;
- корневую папку данных;
- чтение;
- запись;
- обязательные каталоги;
- базовую готовность среды.

Это первая базовая задача приложения.

### `cbr_archiver`

Первая прикладная задача.

Назначение:

- скачать исходные статистические файлы Банка России;
- сохранить результат через активный `FileStore`;
- отдать пользователю понятный путь результата.

### `frg_stage1`

Задача каталогизации FRG-файлов.

Назначение:

- просканировать входную папку;
- построить каталог;
- выбрать актуальные файлы;
- сохранить результат в Excel.

---

## Логика запуска задач

Задача запускается не в GUI-потоке, а в отдельном рабочем потоке.

Это сделано для того, чтобы:

- окно не зависало;
- пользователь видел состояние интерфейса;
- приложение можно было дальше развивать под более длинные сценарии.

Итог выполнения задачи оформляется как `TaskResult`.

В нём содержатся:

- флаг успеха;
- сообщение;
- пути результатов;
- дополнительные детали.

---

## Логи

`app` ведёт собственные логи отдельно от репозитория.

Текущая структура:

```text
%LOCALAPPDATA%\Stratbox\logs\
    app.log
    tasks\
        *.log
```

Это нужно для двух вещей:

- пользователь должен иметь доступ к журналам без поиска по репозиторию;
- обновление Git-репозитория не должно смешиваться с runtime-логами приложения.

На текущем этапе GUI построен как минимальный рабочий каркас на `PySide6`.

Главное окно уже разделяет основные блоки:

- профиль среды;
- статус профиля;
- список задач;
- описание задачи;
- параметры задачи;
- лог выполнения;
- сервисные кнопки;
- строку версии и состояния репозитория.

GUI **не должен** становиться местом, где живёт доменная логика.

Это строго интерфейсный слой.

## Что считается корректным развитием `app`

### Разрешено

Развивать в `app`:

- профили среды;
- диагностику;
- визуальный слой;
- реестр задач;
- сервисные действия;
- обработку пользовательских параметров;
- отображение логов и статусов;
- интеграцию с будущим launcher.

### Не нужно переносить в `app`

Не нужно тащить в `app`:

- доменные преобразования данных;
- разбор форм ЦБ;
- логику FRG-каталогизации;
- загрузку и агрегацию escrow;
- прикладные вычисления и преобразования таблиц.

Это всё должно жить в `stratbox`.

---

## Правила разработки

### 1. `app` не должен ломать `stratbox`

`app` — это слой поверх библиотеки, а не её замена.

### 2. GUI не должен знать внутренности доменного кода

Если для запуска сценария нужно много преобразований параметров — это делается в adapter.

### 3. Пользовательские настройки не должны жить в репозитории

Все runtime-настройки, логи, cache и временные данные должны быть вынесены из Git-дерева.

### 4. Задачи должны быть одинаково запускаемы из GUI и без GUI

Целевой принцип:

- одна и та же прикладная логика должна запускаться:
  - из GUI;
  - из консоли;
  - в будущем через scheduler;
  - в будущем через launcher.

### 5. `app` должен быть готов к роли основного desktop-приложения

Даже если часть поведения пока не доведена, архитектура должна закладываться сразу чисто.

Обратная совместимость здесь не является ограничением.

---

## Что ещё не считается готовым

На текущем этапе `app` не следует считать финальным пользовательским приложением.

Пока ещё не доведены до окончательного состояния:

- tray и жизненный цикл окна;
- launcher-интеграция;
- ручное обновление репозитория из GUI;
- потоковый вывод лога задач в интерфейс;
- полноценное подключение всех сценариев `stratbox`;
- визуальная часть;
- watcher-задачи;
- scheduler-интеграция.

Это нормально. `app` сейчас — базовый каркас, а не финальный продукт.

## Ближайшее направление развития

Следующие естественные шаги:

1. довести профили среды до более гладкого поведения на первом запуске;
2. улучшить открытие результатов и работу с относительными путями;
3. подключить следующие прикладные задачи;
4. довести UX главного окна;
5. подготовить стабильный контракт запуска для будущего `Launcher.exe`.
